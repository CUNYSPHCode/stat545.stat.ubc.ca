Setup

While you used your assignment repository for assignment-1b, just putting your work in the assignment-1b directory, you may choose not to use this repository for this assignment.

For this assignment, you will be developing an R package, ideally open source. Because of this, it is ideal for you to have a separate repository for this assignment. You will be creating a new, separate repository under your own personal GitHub. Using your own personal GitHub will give you more control over the future of this assignment, should you want to develop your R package any further. It will also allow others to install your package more easily using the devtools::install_github() command.

  • Create a new, public repository under your own personal GitHub account, naming it after your R package

Tidy Submission (2 points)

Follow these steps to submit your work. Be sure to familiarize yourself with the rubric for a tidy submission below, before doing these steps.

  • Update your main README for your assignment repository with a link to your new R package repository. Your new repository should not be named assignment-2b. In fact, it should be named whatever you decide to name your package!
  • Make a README for your new package repository, using usethis to kickstart the README, and devtools to render it. Your README should describe what the package is, how to install it, and include an example usage of the package.
  • Tag a release in your GitHub repository corresponding to the version release of your package. Use semantic versioning conventions. While you will be submitting this release link to Canvas, do NOT treat this release like a task for your assignment. This is a release for your package.

Exercise 1: Create an R Package (3 points)

  • Create an R package that wraps at least 1 object. You can wrap more than one object if you wish. To start, use the create_package() and use_git() function to create your R package GitHub repository (if you haven’t made or forked one yet). Make sure that your repository is public so that we can mark it.

Suggested R packages:

  • Turn your function(s) from assignment-1b into an R package
  • Make a data package, like palmerpenguins or gapminder
    • This task is recommended if there is a dataset you are passionate or invested in
  • Add to the vincenzo/powers or jennybc/foofactors packages
    • The easiest way to do this task is to fork the repository you choose

Exercise 1.1: Explain Your Development (2 points)

  • Briefly explain how you made each part of your R package. This doesn’t need to be very detailed, but rather at a high-level
    • Show the exact code you used from usethis or devtools to implement the package component
    • Show any manual work you did with your files

Exercise 1.2: Tests (3 points)

  • Create tests for your package, in the appropriate files and folders, using usethis and testthat.
    • Make sure your tests are also well-documented, meaning that the description/test name is informative, and that any necessary code commenting is present

Exercise 1.3: Documentation of your R package (10 points)

  • Create documentation for your R package.

Specifically:

  • Use roxygen2 comment style #' and tags like @param to document your function(s) or dataset(s)
    • Use devtools::document() to update documentation, so that the ?your_object command is able to fetch your documentation
    • Make sure that all objects are @exported unless there is a valid reason for them to be internal
      • Remember to justify your decisions
  • Document all exported functions and datasets
  • List all package dependencies in their DESCRIPTION file using usethis
  • DESCRIPTION file is otherwise up-to-date and informative
  • If applicable, modify the source code of your function so that functions from other packages are called appropriately (e.g. using ::)
  • Add a license, using usethis
    • Explain why you chose that particular license
  • Make a short vignette for your package

Optional:

  • Add a code of conduct, using usethis::use_code_of_conduct()
  • Make a package website, and activate it with GitHub Pages. I believe it’s really straight-forward with the pkgdown package.
    • use the pkgdown package, using usethis::use_pkgdown(), and then pkgdown::build_suite()
  • Add to the Changelog or add a News article for the ‘release’ of your package
  • Add “badges” to their README

Rubric:

  1. Are all the components of an R package present and high quality, complete with documentation and tests?
  2. Is your R package made using a modern and robust way, using usethis and devtools packages? You should explain your development process in Exercise 1.2.
  3. Your package passes the check() without errors (warnings and notes are OK).
