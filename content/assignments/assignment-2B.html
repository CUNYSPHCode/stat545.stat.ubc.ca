Tidy Submission (2 points)

Follow these steps to submit your work. Be sure to familiarize yourself with the rubric for a tidy submission below, before doing these steps.

  • Update your main README for your assignment repository with a link to your new R package repository. Your new (or forked) repository should not be named assignment-2b. In fact, it should be named whatever you decide to name your package!
  • Make a README for your new package repository, using usethis to kickstart the README, and devtools to render it. Your README should describe what the package is, how to install it, and include an example usage of the package. The datateachr repository is a good example for a R dataset package. The distplyr repository is a good example for an R function(s) package.
  • Tag a release in your R package repository corresponding to the version release of your package. Use semantic versioning conventions. This is a release for your package.
    • Semantic versioning conventions: MAJOR.MINOR.PATCH, where initial development starts at 0.1.0, and then increment the minor version for each subsequent release
      • Only release 1.0.0 if your package is being used in production and ready for public use
      • e.g. 0.1.0 :arrow_right: 0.1.1 :arrow_right: 0.1.X :arrow_right: 1.0.0 :arrow_right: 1.0.1 :arrow_right: 1.0.X :arrow_right: 1.1.0
    • Your release versioning is especially important if you will be choosing to develop your package further in Assignment 5-B
  • Tag a release for your assignment repository, e.g. stat-545b-assignments-GitHub-Username for your assignment 2-B tagged release to Canvas

To clarify, you will be making two tagged releases for this assignment:

  1. A semantic versioning tagged release for your R package repository (most likely 0.1.0)
  2. A tagged release for your assignment repository (i.e. stat-545b-assignments-GitHub-Username) that you will be submitting to Canvas (most likely hw2.0 or a2.0, or whichever is consistent with your assignment 1-B tagged release)

Exercise 1.3: Documentation of your R package (10 points)

  • Create documentation for your R package.

Specifically:

  • Use roxygen2 comment style #' and tags like @param to document your function(s) or dataset(s)
    • Use devtools::document() to update documentation, so that the ?your_object command is able to fetch your documentation
    • Make sure that all objects are @exported unless there is a valid reason for them to be internal
      • Remember to justify your decisions
  • Document all functions and datasets
  • List all package dependencies in their DESCRIPTION file using usethis
  • DESCRIPTION file is otherwise up-to-date and informative
  • If applicable, modify the source code of your function so that functions from other packages are called appropriately (e.g. using ::)
  • Add a license, using usethis
    • Explain why you chose that particular license, adding to EXPLANATIONS.md (see Exercise 1.1 example above)
  • Make a short vignette for your package

Optional:

  • Add a code of conduct, using usethis::use_code_of_conduct()
  • Make a package website, and activate it with GitHub Pages.
    • use the pkgdown package, using usethis::use_pkgdown(), and then pkgdown::build_site()
  • Add to the Changelog or add a News article for the ‘release’ of your package
  • Add “badges” to your README
